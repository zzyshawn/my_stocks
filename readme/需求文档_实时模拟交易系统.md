# 股票模拟交易系统 - 实时模拟交易需求文档

## 1. 背景与目标

当前项目大部分功能已实现，但“实时模拟”这一块实现偏离需求，主要问题见：
- 配置与数据目录：`config/config.yaml:18-40`, `config/config.yaml:316-353`
- 目录结构：`readme/tree.md:9-49`
- 现有模拟实现：`src/simulation/data_preparation.py:32-44`, `src/simulation/data_preparation.py:321-395`
- 现有实时命名参考：`src/data/realtime/my_real_time.py:197-219`

本次目标不是重写分析算法，而是重做数据准备与实时模拟编排层，使其满足：

1. 对指定股票按测试日生成“模拟实时K线”
2. 逐根5分钟K线驱动 5m / 30m / 日线同步增量生成
3. 每一步都完成 realtime 落盘、与 `_1` 基线拼接、去重、回写 backtest
4. 复用已有缠论与技术指标函数，生成实时分析结果
5. 为后续 Agent 模拟交易、历史结果评估预留接口
6. 输出一套可执行的数据准备完整流程图
7. 新增统一 orchestrator 主入口，支持 `simulation` / `realtime` 两种模式
8. 每个 5 分钟步内，在 `d/30/5`、缠论、技术指标全部产物生成后，打包 JSON 并发送 `DATA_READY` 通知

## 4. 现有实现问题总结

现有 `src/simulation/data_preparation.py` 存在以下偏差：

1. **交易时间不符合新需求**
   - 现实现含 `09:30`、`13:00`：`src/simulation/data_preparation.py:33-43`
   - 新需求应从 `09:35`、`13:05` 开始

2. **输出位置不符合需求**
   - 现实现将每日快照输出到 `backtest/{date}/`：`src/simulation/data_preparation.py:321-395`
   - 新需求要求：
     - realtime 快照存 `realtime/`
     - 拼接后的全量结果存 `backtest/`

3. **输出形式不符合需求**
   - 现实现按索引输出大量散文件：`rt_5_0.xlsx`、`rt_30_0.xlsx`
   - 新需求要求：
     - 保持与现有 realtime 命名规则兼容
     - 同时保留一份**无时间戳主文件**
     - 时间戳调试文件可关闭

4. **30分钟/日线生成逻辑表达不完整**
   - 新需求明确：**1根5分钟K线也可以临时生成30分钟线和日线**
   - 即不是“只有满6根才生成”，而是“每一步都生成当前累计态”

5. **缺少每步拼接**
   - 新需求要求每新增一根5分钟，就执行一次 `_1 + realtime` 拼接、去重、回写 backtest

## 5.6 每步 JSON 打包要求

每个 5 分钟步结束后，都要把当前步分析结果打包为一个 JSON 文件。

### 保存目录
- realtime：`H:/股票信息/监控清单/test_data/real`
- simulation：`H:/股票信息/监控清单/test_data/backtest`

### 文件粒度
- 每个 5 分钟步生成一个 JSON 文件
- 通知中的 `data_path` 指向该步 JSON 文件

### 顶层结构
```json
{
  "stock_code": "603906",
  "day": {},
  "min30": {},
  "min5": {}
}
```

### 周期结构要求
每个周期包含：
- `kline_range`: `[最低, 最高]`
- `time`: 最后 10 个时间戳
- `close`: 最后 10 个收盘价
- `value`: 最后 10 个成交量（即 volume）
- `chanlun`
- `indicators`

#### day
- `kline_range` 取最近 10 根 K 线的最低 ~ 最高
- `chanlun.fx_*` 取最后 10 个分型
- `chanlun.bi_*` 取最后 6 个笔顶点

#### min30
- `kline_range` 取最近 20 根 K 线的最低 ~ 最高
- `chanlun.fx_*` 取最后 10 个分型
- `chanlun.bi_*` 取最后 8 个笔顶点

#### min5
- `kline_range` 取最近 10 根 K 线的最低 ~ 最高
- `time` / `close` / `value` 取最后 10 个数据

### 缠论字段
- `zs_range`: `[ZD, ZG]`
- `fx_times`
- `fx_types`
- `bi_times`
- `bi_types`
- `bi_values`

说明：
- `bi_values` 根据笔顶点类型取值
- 顶分型取 `high`
- 底分型取 `low`

### 技术指标字段
- 放入 `indicators` 对象
- 每个指标保存最后 10 个值
- 当前已补入主要指标：
  - `MA10/20/60/120`
  - `RSI_L6`
  - `RSI_H6`
  - `MACD_DIF`
  - `MACD_DEA`
  - `MACD_BAR`
  - `VOL`
  - `换手率`
  - `量比`
  - `VOL_MA5`
  - `VOL_MA20`
  - `VWAP`
- 数值格式规则：
  - 价格统一两位小数
  - 指标按有效位压缩
  - 大于 10 的指标当前按整数输出

## 7. 单日逐根模拟规则

设当天 48 根 5分钟K线为 `bar[1] ... bar[48]`。

每处理一根 `bar[i]`，都必须完成以下动作。

### 7.1 生成当前 realtime 5分钟数据
将当日前 `i` 根 5分钟数据组成当前实时5分钟数据集：

- 第1步：只有第1根
- 第2步：前2根
- ...
- 第48步：完整48根

保存到：
- `realtime/{code}_rt_5.xlsx`
- 调试模式可另存时间戳版

### 7.2 生成当前 realtime 30分钟数据
30分钟线由 5分钟线累计生成。

规则：
- 每 6 根 5分钟K线构成一个完整30分钟区间
- 但**未满 6 根时，也必须生成一个“当前累计中的30分钟线”**
- 因此，第1根5分钟也可以生成30分钟线，第2根也可以，直到第6根形成该区间完整30分钟线
- 第7根开始进入第2个30分钟区间，同样累计生成

#### 具体要求
- 1~6 根 5m：realtime 中保持 1 根 30m
- 7~12 根 5m：realtime 中保持 2 根 30m
- 13~18 根 5m：realtime 中保持 3 根 30m
- ...
- 43~48 根 5m：realtime 中保持 8 根 30m

#### 30分钟线聚合规则
对于某个30分钟区间：
- 开盘 = 区间第一根5m开盘
- 最高 = 区间内最高价最大值
- 最低 = 区间内最低价最小值
- 收盘 = 区间最后一根5m收盘
- 成交量 = 区间内成交量求和

#### 时间要求
- 30分钟线必须只使用**当天**的5分钟数据
- 不允许跨天累计

保存到：
- `realtime/{code}_rt_30.xlsx`
- 调试模式可另存时间戳版

### 7.3 生成当前 realtime 日线数据
日线也由当天累计的 5分钟线实时聚合得到。

规则：
- 第1根5分钟即可生成临时日线
- 第48根时得到完整日线

聚合规则：
- 开盘 = 当天第一根5m开盘
- 最高 = 当天截至当前最高价
- 最低 = 当天截至当前最低价
- 收盘 = 当前最后一根5m收盘
- 成交量 = 当天截至当前成交量总和

保存到：
- `realtime/{code}_rt_d.xlsx`
- 调试模式可另存时间戳版

## 8. 每一步拼接规则

这是本次需求最核心的部分。

每生成一次 realtime 数据，都必须立即拼接一次 backtest 数据。

### 8.1 拼接来源
对于每个周期（5/30/d）：

- 历史基线：`backtest/{code}_{period}_1.xlsx`
- 当前实时：`realtime/{code}_rt_{period}.xlsx`

### 8.2 拼接目标
生成：
- `backtest/{code}_{period}.xlsx`

### 8.3 拼接规则
1. 读取 `_1`
2. 读取 realtime 当前文件
3. 行拼接
4. 按“日期”去重
5. 按时间升序排序
6. 保存回 `backtest/{code}_{period}.xlsx`

### 8.4 去重规则
建议按以下字段去重：
- 主键优先：`日期`
- 若存在同一 `日期` 多条记录：
  - 保留后者（即 realtime 最新值覆盖旧值）

这样可以确保：
- 基线历史不重复
- 当日增量K线在回写时始终以当前最新模拟值为准

### 8.5 执行频率
每新增 1 根 5分钟K线，就要执行 3 次拼接：
- 拼接 5分钟
- 拼接 30分钟
- 拼接 日线

即每一步都得到一套“当前时点完整可见”的 backtest 数据。

### 9.4 real 参照数据生成规则
- 数据源必须使用股票根目录下的全量原始 K 线文件：
  - `{code}/{code}_d.xlsx`
  - `{code}/{code}_30.xlsx`
  - `{code}/{code}_5.xlsx`
- 不使用 `_1/_2` 作为 `real` 数据源
- 先基于全量原始数据生成三周期缠论与技术指标结果，输出到：
  - `backtest/out/real/d/`
  - `backtest/out/real/30/`
  - `backtest/out/real/5/`
- 后续在测试验证时，再从 `real` 结果中按测试交易日或步数提取对照范围

### 9.5 test 模拟数据生成规则
- 数据源使用模块2每一步拼接后的 backtest 文件：
  - `backtest/{code}_5.xlsx`
  - `backtest/{code}_30.xlsx`
  - `backtest/{code}_d.xlsx`
- 每一步生成新的 backtest 数据后，不直接对全量数据整体输出，而是先按配置截取测试视图窗口，再计算三周期技术指标与缠论结果
- 截窗规则放在 `config/config.yaml -> analysis.runner.lookback_window`：
  - `trading_days` 默认 `30`
  - `d` 保留 `30 * 1` 根
  - `30` 保留 `30 * 8` 根
  - `5` 保留 `30 * 48` 根
- 主结果输出到：
  - `backtest/out/test/d/`
  - `backtest/out/test/30/`
  - `backtest/out/test/5/`
- 若开启 debug，则每一步过程结果额外保存到：
  - `backtest/out/test/debug/{trade_date}/step_01/ ... step_48/`
- 主文件允许覆盖，debug 文件用于保留过程快照，便于验证

### 9.6 验证口径
- `test` 结果用于表示“当前模拟时点可见分析结果”
- `real` 结果用于提供真实全量参照基线
- 验证时按测试日期、周期、步数从 `real` 中提取对应范围，与 `test` 同周期结果对比
- 默认保留 `test` 主文件，debug 过程文件按配置决定是否保留

## 4. 模拟生成单日实时K线需求

### 4.1 重复过程2.2~3
- 对某个测试交易日，按当天48根5分钟线逐根模拟生成 realtime 5m / 30m / 日线
- 每合成一次 realtime 数据，立即执行一次过程3（缠论与技术指标生成）
- 每合成一次 realtime 数据，立即执行一次 backtest 拼接
- 模块2除自动校验外，还要生成 1~48 步文件供人工验收

### 11.1 输入
- 股票代码
- 测试日期列表
- 拆分后的 `_1/_2` 数据

### 11.2 执行逻辑
对测试区间内每个交易日，重复执行：
1. 取出该日48根5分钟K线
2. 从第1根开始逐根模拟
3. 每根都生成 realtime
4. 每根都拼接回 backtest
5. 每根都调用指标/缠论
6. 完成当天48步后，调用独立 Agent 模块完成当天判断/验证
7. 使用当天真实数据做日终评估
8. 仅当当天评估验收通过后，才允许进入下一交易日

### 11.3 跨日规则
- 第 N 日结束后，生成该日最终的 backtest 全量文件
- 第 N+1 日开始前：
  - backtest 中上一日已经成为新的历史可见数据
- 但 `_1` 作为初始基线文件不改写
- 推荐以“当前 backtest 全量文件”作为下一步连续模拟输入视图

## 9. 配置与目录依据

### 13.1 5分钟数据校验
- 当日必须为 48 根
- 时间序列必须匹配交易时段
- 不允许跨天混入
- 日期列需可解析为 datetime

### 13.2 30分钟数据校验
- 第 `i` 步应有 `ceil(i / 6)` 根30分钟线
- 每根30分钟线只来自当天
- 第 6、12、18...48 步时，对应最后一根30分钟线为“完整确认状态”

### 13.3 日线校验
- 每步必须恰好有 1 根当日日线 realtime 数据
- 最终第48步生成完整当日日线

### 13.4 拼接结果校验
- `backtest/{code}_{period}.xlsx` 行数 = `_1` + 当前 realtime 去重后结果
- 时间升序
- 不重复
- 当天最后一条记录与 realtime 最后一条一致

## 16. 模块化串行验收原则

必须遵循：**一个模块一个模块验证验收，通过后才能开发下一个模块。**

### 模块1 数据拆分验收
- 现有 `_1/_2` 历史产物不作为基线
- 直接以原始 `603906_d.xlsx`、`603906_30.xlsx`、`603906_5.xlsx` 重新拆分结果验收
- 5min 可以与原始数据逐行直接对照
- 30 和日线做部分人工对照，重点看测试日边界、首尾记录、条数

### 模块2 单日实时K线与拼接验收
- 自动验收：程序自动校验 1~48 步每一步的 realtime 与拼接 backtest 的行数、去重、时间升序、30m 根数递增、日线唯一性
- 人工验收产物：保留 1~48 步对应文件，供人工逐步检查
- 不仅抽查 1/6/7/12/48，要求全量生成 1~48 步文件

### 模块3 指标/缠论验收
- 基于模块2每步拼接结果执行
- 检查关键周期都能产出完整结果
- 最后一步结果应与全量 backtest 数据一致

### 模块4 单日 Agent + 日终评估验收
- Agent 模块独立
- 必须在单日48步模拟完成后再运行
- 使用当天真实数据做验证与日终评估
- 验收通过后，才能进入下一天或下一模块

### 模块5 多日循环验收
- 严格按“单日模拟 → Agent → 日终评估 → 通过再下一日”执行
- 任一日未通过，不进入下一日

## 17. 总体验收标准

1. 能读取目标股票目录并自动创建 `realtime/`、`backtest/`
2. 能将 `d/30/5` 正确拆分为 `_1/_2`
3. 能从测试日当天 48 根 5分钟线逐根模拟
4. 每一根 5m 都能同步生成当前 30m 和日线
5. 每一步都能将 `_1 + realtime` 拼接成 `backtest/{code}_{period}.xlsx`
6. 拼接结果无重复、按时间排序、数据正确
7. 每一步都可触发技术指标/缠论生成
8. 单日48步完成后，Agent 独立模块可基于真实数据完成当天判断与验证
9. 日终评估完成且通过后，系统才允许进入下一交易日
10. 多日模拟必须遵循“一天一验收”的串行流程
11. 支持关闭时间戳调试文件，仅保留无时间戳主文件
