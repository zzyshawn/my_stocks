# 股票模拟交易系统 - 实时模拟交易开发计划

## 1. 开发原则

1. 一个模块一个模块开发
2. 一个模块一个模块验收
3. 当前模块未通过，不开发下一个模块
4. 多日模拟必须在“单日模拟 + Agent + 日终评估”通过后再推进下一天
5. Agent 模块独立，后续支持实时验证扩展

## 2. 模块划分

### 模块1 数据拆分
状态：已完成

职责：
- 读取 `603906_d.xlsx`、`603906_30.xlsx`、`603906_5.xlsx`
- 按测试起始日拆分为 `_1/_2`
- 输出到 `{code}/backtest/`

建议文件：
- `src/simulation/splitter.py`
- `tests/test_splitter_603906.py`

输入：
- 原始 `d/30/5` 文件
- 测试起始日配置

输出：
- `backtest/{code}_d_1.xlsx`
- `backtest/{code}_d_2.xlsx`
- `backtest/{code}_30_1.xlsx`
- `backtest/{code}_30_2.xlsx`
- `backtest/{code}_5_1.xlsx`
- `backtest/{code}_5_2.xlsx`

验收方式：
- 现有 `_1/_2` 历史文件不作为基线
- 以重新从原始数据拆分的结果验收
- 5min 与原始数据逐行对照
- 30min 和日线做人工边界对照

通过条件：
- 边界正确
- 条数正确
- 首尾记录正确
- 文件命名正确

### 模块2 单日实时K线与拼接
状态：进行中（基础功能完成，逐步打印与人工验收优化中）

职责：
- 从 5分钟 `_2` 中取某测试日 48 根数据
- 按 09:35 / 13:05 规则逐根模拟
- 生成 realtime 5/30/d
- 每步执行 `_1 + realtime` 拼接回 backtest
- 自动校验每一步数据正确性
- 生成 1~48 步文件供人工验收

建议文件：
- `src/simulation/realtime_generator.py`
- `src/simulation/backtest_merger.py`
- `tests/test_realtime_generator_603906.py`

输入：
- `backtest/{code}_5_1.xlsx`
- `backtest/{code}_5_2.xlsx`
- `backtest/{code}_30_1.xlsx`
- `backtest/{code}_d_1.xlsx`
- 指定测试交易日

输出：
- `realtime/{code}_rt_5.xlsx`
- `realtime/{code}_rt_30.xlsx`
- `realtime/{code}_rt_d.xlsx`
- `backtest/{code}_5.xlsx`
- `backtest/{code}_30.xlsx`
- `backtest/{code}_d.xlsx`
- 调试/人工验收用 1~48 步文件

自动验收：
- 程序自动检查 1~48 步
- 每一步检查：
  - realtime 行数正确
  - backtest 行数正确
  - 日期升序
  - 去重正确
  - 30分钟根数 = `ceil(step / 6)`
  - 日线始终为 1 根当日数据

人工验收：
- 必须保留 1~48 步文件
- 供人工逐步核对，不只抽查若干步

通过条件：
- 自动校验全通过
- 1~48 步文件完整生成
- 人工抽看关键步与预期一致

### 模块3 指标/缠论 + JSON 打包 + orchestrator 通知
状态：进行中（已完成结构化结果摘要、JSON 打包、simulation orchestrator 骨架）

职责：
- 基于模块2每一步拼接后的 backtest 数据，生成模拟分析结果
- 基于根目录全量原始 K 线数据，生成参照分析结果
- 调用技术指标与缠论分析
- 将结果统一输出到 `backtest/out/`，供后续 Agent 使用与人工验证
- 将当前步分析结果打包成 JSON，保存到监控目录
- 在每步全部分析产物生成后发送 `DATA_READY` 通知
- 提供 orchestrator 编排入口，统一 simulation / realtime 模式

建议文件：
- `src/simulation/analysis_runner.py`
- `src/simulation/orchestrator.py`
- `tests/test_analysis_runner_603906.py`
- `tests/test_orchestrator_message_schema.py`
- `tests/test_orchestrator_simulation_603906.py`

复用入口：
- `src/analysis/indicators.py`
- `src/python-biduan/stock_analysis.py`

输入：
- real：`{code}/{code}_5.xlsx`、`{code}_{30}.xlsx`、`{code}_d.xlsx`
- test：每步 `backtest/{code}_5.xlsx`、`backtest/{code}_30.xlsx`、`backtest/{code}_d.xlsx`
- 配置文件中的指标开关、参数、debug 开关、输出目录配置
- 配置文件中的 test 截窗规则：`analysis.runner.lookback_window`

输出：
- `backtest/out/real/d|30|5/` 下的参照指标结果与缠论结果
- `backtest/out/test/d|30|5/` 下的模拟指标结果与缠论结果
- `backtest/out/test/debug/{trade_date}/step_01...step_48/` 下的过程结果（可配置开启/关闭）
- `H:/股票信息/监控清单/test_data/backtest/` 下每步一个 JSON 文件
- `H:/股票信息/监控清单/test_data/real/` 下每步一个 JSON 文件
- 每步对应一个 `DATA_READY` 通知请求体

验收方式：
- 检查 `5m/30m/d` 三个周期都可产出
- 检查 `real` 使用的是根目录全量源数据，不使用 `_1/_2`
- 检查 `test` 使用的是模块2当前拼接 backtest 数据
- 检查 `test` 输出前已按配置截窗：日线 `30*1`、30分钟 `30*8`、5分钟 `30*48`
- 检查 debug 开启时能保留逐步产物
- 检查验证时可从 `real` 中按测试日期或步数提取对照范围

通过条件：
- 三周期结果完整
- `real/test` 目录结构正确
- 主文件与 debug 文件行为符合配置
- 最新 `test` 结果与最新 backtest 数据一致

## 3. 开发顺序

### Phase 1
先实现模块1，不碰模块2~5。

### Phase 2
模块1验收通过后，实现模块2。

### Phase 3
模块2验收通过后，实现模块3。

### Phase 4
模块3验收通过后，实现模块4。

### Phase 5
模块4验收通过后，实现模块5。

## 4. 模块2特别说明

模块2不能只做抽查式验收，必须同时满足：

1. **自动验收全 1~48 步**
2. **生成 1~48 步文件供人工验收**

建议额外保留：
- `realtime/steps/step_01/`
- `realtime/steps/step_02/`
- ...
- `realtime/steps/step_48/`

每步至少保存：
- 当前 realtime 5
- 当前 realtime 30
- 当前 realtime d
- 当前拼接后 backtest 5
- 当前拼接后 backtest 30
- 当前拼接后 backtest d

当前指定真实验证股票代码：`603906`

模块验证要求：
- 模块1：使用 603906 原始数据重建拆分结果
- 模块2：使用 603906 的真实某个交易日做 48 步实时模拟

- 模块4：使用 603906 单日真实数据做 Agent 验证与日终评估
- 模块5：使用 603906 进行多日串行模拟

1. 模块1~5 均有独立输入、输出、验收口径
2. 每个模块通过后才进入下一模块
3. 模块2支持自动+人工双重验收
4. 单日完成后必须先 Agent、再评估、再下一天
5. 文档与实现保持一致
